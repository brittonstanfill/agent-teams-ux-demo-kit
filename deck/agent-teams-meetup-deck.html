The thesis
Agent teams earn their tokens for three specific shapes of problem.
Everything else, reach for a subagent.
01
Horizontal context
The work doesn't fit in one window — even a 1M one. Five teammates at 50% beat one session at 90%.
02
Restartable specialists
The run has to survive overnight. Each teammate is a real Claude session — if one's window fills, spawn a fresh one.
03
Adversarial loops
Tester finds a failure, developer fixes it, reviewer signs off — without you. Subagents report back and die. Teammates persist.

This is the spine of the whole talk. Three legs. Repeat the three labels — context, specialists, loops — because you'll come back to them.

Important framing: everything else, use a subagent. This is what makes the talk credible. You're not selling agent teams as a universal upgrade — you're saying they're a power tool for a narrow set of problems.

If someone has heard the "agent teams = parallel subagents" framing, this is where you correct it: "Parallelism is incidental. The value is in the three problem shapes."

Subagents vs. agent teams
Subagents work for you.
Teammates work with each other.

This is the conceptual heart of the talk. If someone takes notes, this is the slide they take notes on.

Land the contrast hard: subagents are workers. Teammates are peers. The product of a subagent is a summary; the product of an agent team is whatever the team builds together.

If there's time, name an example for each: subagent — "find every file that imports lodash and tell me the count." Agent team — "four investigators, each with a different theory, argue until consensus."

Reach for a team when…
  • The work won't fit in one context window. Cross-codebase refactors, full-codebase audits, 30-file features.
  • The run has to survive overnight. Long migrations, autonomous test sweeps, multi-hour investigations.
  • You need handoffs you don't supervise. Tester → developer → reviewer loops. Competing-hypothesis debates.
  • Each lens needs its own context. Security review, perf review, test review — three full sessions beats one session juggling three modes.

Each bullet maps to one of the three legs (or a clean extension of leg 3). Don't read all four — pick the two that resonate most with your room.

For a designer-heavy room: lean on bullet 4 (parallel review with distinct lenses) and bullet 3 (handoff loops). The audit pattern translates directly to design critique.

…and when not to
  • The work is sequential. A blocks B blocks C.
  • Two teammates would edit the same file.
  • The task fits comfortably in one session.
  • "First plausible answer" is fine — you're not stress-testing anything.
  • You're using the team as orchestration — that's what subagents are for.
Honest framing: most everyday coding work is in this list.

This slide is your credibility move. By naming when NOT to use the tool, you make the case for when TO use it sound like considered advice, not evangelism.

The "two teammates editing the same file" point is in the docs explicitly — last write wins, no locking. Worth saying out loud.

More teammates isn't free

Communication channels grow as n(n−1)/2 — exponentially, not linearly. The same math that breaks human meetings breaks agent teams.

3
3 channels
A specialist, a challenger, and a lead. Everyone tracks everyone. This is the sweet spot.
15
6 teammates → 15 channels
Still workable, but you're now paying real reconciliation cost. The 07- debate prompt sits here on purpose.
45
10 teammates → 45 channels
Groupthink, dropped threads, drift. Beyond ~8 you're paying for coordination, not coverage.
The "rule of 7" from meeting research applies here too: 5–8 teammates for decisions, fewer is better. Add lenses, not headcount.

This is the counterweight to the agent-team hype. Adding agents has the same n(n−1)/2 cost curve as adding people to a meeting — Brooks's law, basically.

Land the takeaway: the 07- debate prompt deliberately caps at 3–4 specialists + devil's advocate. That's 4–5 teammates, ~10 channels. Resist the urge to spawn all 8 agents at once.

If asked "why not just run all 8 UX agents?" — that's 28 channels. The disagreement stops being signal and starts being noise.

What it is
A terminal multiplexer. One window, many panes, sessions that survive a closed lid. Made in 2007 for exactly this kind of split-attention work.
Why it matters for agent teams
Four teammates running in parallel = four streams of output. Splitting into panes lets you watch the debate happen instead of scrolling backwards trying to reconstruct it.
The seven commands you need
brew install tmux
One-time install. apt / dnf on Linux.
tmux new -s teams
Start a session named teams. Run claude inside. Works in any terminal.
tmux -CC attach -t teams
iTerm2 only. Renders each pane as a native iTerm2 tab/split — real scrollback, search, copy/paste. Same tmux server underneath.
Ctrl-b "   Ctrl-b %
Split horizontal (") or vertical (%). Repeat for four panes.
Ctrl-b ←/→/↑/↓
Move between panes without touching the mouse.
Ctrl-b d
Detach. Session keeps running — close the lid, come back later.
tmux attach -t teams
Reattach. Everything is exactly where you left it.
-CC vs plain — when to pick which
tmux -CC — iTerm2 only. Native windows on a projector, easy scrollback, best for the live demo or screen recording. Loses the visual when you switch terminals or SSH out.
tmux new -s — works everywhere (Terminal.app, Ghostty, Alacritty, Warp, over SSH). Durable detach/reattach across hours or days. The canonical setup the Claude Code agent-teams docs assume.
Both talk to the same tmux server — -CC is purely an iTerm2 rendering layer over it. Meetup demo on a Mac → -CC. Overnight runs or remote work → plain.

Frame it like this: "Agent teams without tmux is like watching a four-way debate by reading the transcript at 4x speed. Agent teams with tmux is sitting in the room."

The prefix key is Ctrl-b by default. A lot of people remap it to Ctrl-a (matches GNU screen muscle memory). Mention it if anyone asks; don't bring it up unprompted.

If split-pane mode isn't cooperating during the live demo, you can fall back to a single terminal with `claude` in in-process mode — the team still works, you just narrate instead of letting the audience see all four at once. Don't burn time troubleshooting tmux on stage.

On macOS: brew install tmux. On Ubuntu/Debian: sudo apt install tmux. Iterm2 and Ghostty both have native split panes too, but tmux is the one that lets the session survive a disconnect — handy when your meetup wifi drops.

-CC vs plain: tmux -CC is iTerm2's Control Mode — iTerm2 becomes the tmux client and renders each pane as a real OS-level tab or split, with native scrollback, search, mouse, and themes. Best for the live meetup (Mac + iTerm2 + projector) and for screen recordings. Plain tmux new -s teams works in any terminal and survives disconnects, so it's the right call for SSH, overnight runs, or anything you need to reattach to hours later. Both connect to the same tmux server — -CC is purely an iTerm2 rendering layer over it. Claude Code's teammateMode: "tmux" doesn't care which mode you use.

The mechanism
The arguing isn't the show.
It's the design.
01
Specialists clash by design
Each agent's domain touches another's. Behavioral fights Content on framing. A11y fights Visual on contrast. The conflicts are the data — averaged opinions hide the real tradeoffs.
02
Positions, not averages
Sub-agents stack findings. Teams take defended positions — "we delayed the 9:47 PM push, here's the cognitive evidence; we removed 'abnormal' from the lock screen, here's why."
03
Honest handoffs
Tensions that can't resolve get handed back to the human lead with the contending positions stated cleanly. That handoff is itself a deliverable — only the team pattern produces it.

Anchor on the headline: the arguing isn't the show, it's the design. The visible spectacle of agents messaging each other isn't a gimmick — it's the mechanism by which a defended design emerges.

The contrast: a sub-agent stack converges on bland because no specialist defended a contested position. The agent team takes positions, justifies them, and surfaces what can't be resolved.

If asked "couldn't I do all this with one agent in one prompt?" — yes, but you'd get coherence, not debate. The debate log is the artifact the single-prompt version can't produce.

Three tips that will save your weekend
01
Pre-approve permissions before you spawn.
Every permission prompt bubbles up to the lead and freezes the whole team. Put your common bash and read patterns in .claude/settings.json before the demo.
02
Decisions in files, not chat.
If two teammates need to know it, write it down. Chat is for live disagreement only. This kills 80% of "team chat is just reconciliation overhead" critique.
03
Check in every 10 minutes.
Sequential drift is silent. If you see "waiting for X" twice, intervene. Unattended teams burn usage limits fast.

These three are the operational difference between a team that works and a team that wastes 4 hours of your weekly plan.

Tip 1: I'd say "the most underrated improvement to your demo" — pre-approval is the difference between a smooth team run and a permission-prompt soup.

Tip 2: this is the answer to the strongest skeptic argument. Don't expect agents to "discover" decisions through chat. Write them down.

Tip 3: childofsol on r/ClaudeCode lost a 5-hour usage window because he didn't check in. Don't be that person.

Bonus · computer-use watchdog
paste into Codex / Claude computer use
Keep the team unblocked while you're away from the keyboard.
Runs every 5 minutes. Watches the tmux session, approves safe prompts, nudges stalled agents, escalates anything risky.
You are a watchdog operator for a Claude Code agent team running in a tmux session. Your job is to keep the team unblocked while a human is away. You can see the screen and use mouse + keyboard. Loop every 5 minutes. Each cycle: 1. Take a screenshot of the active terminal window. If the team is running in tmux, capture each visible pane. 2. Classify each agent pane into exactly one state: - WORKING — tokens streaming, file edits visible, output changed in the last 60s - WAITING — explicit "waiting for X" or "needs decision from Y" in the last 20 lines - PROMPTED — a permission prompt is on screen (Yes/No, Allow/Deny, 1/2/3) - IDLE — no output change for more than 10 minutes AND no pending question - DONE — explicit completion message ("done", "finished", "wrote final-recommendation.md") - ERROR — visible exception, "session expired", red error text, or "rate limit" 3. Take exactly ONE action per pane, in this order of preference: - WORKING or DONE → do nothing. - PROMPTED → if the prompt matches the allowlist below, click the approve option. Otherwise, leave it untouched and append to watchdog.log: "HUMAN NEEDED ". - WAITING → if the named dependency is in DONE state, click into the waiting pane and type a one-line nudge: "@ has finished — see ; please proceed." then Enter. - IDLE → click into the pane and type: "Status check: are you blocked? If yes, name the blocker and tag the right teammate. If no, continue with your current task." then Enter. - ERROR → capture the last 30 lines to watchdog.log. Do NOT auto-restart. Mark as HUMAN NEEDED. 4. Allowlist — auto-approve ONLY these prompt types: - read access to any path under the project working directory - bash: ls, cat, head, tail, grep, rg, find, git status, git diff, git log, npm test, npm run *, pytest, python -m pytest - file writes inside the project's outputs/ or demo-output/ directory Anything else → HUMAN NEEDED. 5. Hard stops — do NOT act, escalate immediately, and pause the loop: - any prompt containing: rm, sudo, git push, git reset, --force, drop, delete, DROP TABLE - any pane showing "rate limit", "usage limit", "billing", "quota exceeded" - more than 2 panes in ERROR simultaneously - any prompt asking for credentials, API keys, or 2FA codes 6. After each cycle, append one status row to watchdog.log: [ISO-8601 timestamp] WORKING=N WAITING=N IDLE=N PROMPTED=N DONE=N ERROR=N acted=N escalated=N 7. Stop the loop when ANY of these is true: - all panes are DONE - the HUMAN NEEDED list is non-empty for two consecutive cycles - a pane shows the literal text STOP-WATCHDOG - you have run 24 cycles (2 hours total) Rules you never break: - Never edit code files yourself. You only observe and send short messages. - Never approve a prompt outside the allowlist. - Never restart a session, never run claude --resume, never kill a pane. - If you are uncertain about a state classification, choose the more conservative one (treat WAITING as PROMPTED, treat PROMPTED as HUMAN NEEDED). - One action per pane per cycle. If two actions seem warranted, take the safer one and log the other as a suggestion. Begin the first cycle now. Report what you observed and what you did before sleeping until the next cycle.

This is for the "overnight run" scenario. Drop the prompt into Codex with computer-use enabled, or into Claude's computer-use tool, point it at the tmux session, and walk away.

The allowlist and hard-stop sections are the load-bearing part. Without them you've built a footgun — a computer-use model that will happily approve a git push --force at 3am.

The 24-cycle cap is the soft kill switch. If the team can't finish in 2 hours of watchdog babysitting, the team itself is the problem.

Honest framing: this is a workaround for the fact that agent teams can stall on permission prompts. Better long-term answer is to pre-approve permissions in .claude/settings.json (see Tip 1). The watchdog is for the cases the pre-approval list doesn't cover.
