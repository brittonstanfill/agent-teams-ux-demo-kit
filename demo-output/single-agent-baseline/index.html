Trip recovery flow · canceled-flight redesign

A flow that helps a tired traveler get there — and ends with what happens next, not "Done."

Four screens after the wake-up text. The redesign reframes the SMS, drops the misleading default, pulls hotel and meals out of the FAQ, and replaces the dead-end confirmation with a save-to-offline timeline. Mobile-first; supports light and dark; respects reduced motion; tap targets ≥ 44 px.

Screen 0

The wake-up text

If the SMS doesn't say what happened, every screen after it has to apologize for it. The text carries the cause, what tapping does, and the two big things the user is about to decide.

10:46 PM

Design notes

  • Names the event. "Canceled" — not "schedule irregularity." A stressed user shouldn't have to translate jargon to know what's happening.
  • Names the cause. "Crew availability" tells the user this is on the airline, not on weather they're about to check on their own.
  • Names what tapping does. "See your options for tonight and tomorrow" — explicit, concrete, two-horizon.
  • Stays under 160 characters. Renders as a single SMS on every carrier we tested.

SMS is plain text — no rich formatting that breaks on basic phones or screen readers. Link clearly distinguished by underline and color (1.4.1, contrast ≥ 4.5:1).

Design notes

  • No tabs. Tabs hide alternatives and let us ship the wrong default. Three equal-weight cards expose all three paths at once.
  • "Most people start here" — not "Recommended." Recommendations without reasons feel like sales pressure. This phrase is plain, explainable, and not a promise.
  • Entitlement on the first screen. A covered-tonight chip surfaces the hotel/meal coverage before the user worries about it. Reinforces that asking for help is not a fight.
  • Persistent agent affordance. Bottom of every screen, same visual weight. Self-service should be the easy path, not the only path.
  • Status announced once. An aria-live="polite" region announces the cancellation on screen load — and only on load. We do not poll.

Accessibility

Status uses icon + text (not color alone — 1.4.1). All actions ≥ 44 × 44 px (2.5.5). Visible focus rings (2.4.7). Headings in document order (1.3.1).

Design notes

  • Reason tags, not "Recommended." Each tag is a falsifiable claim: "Earliest arrival," "Fewest stops." If they're not true, they don't appear.
  • No fare deltas on like-for-like rebooks. Price is for upgrade decisions, later, with consent. Showing fare deltas during a cancellation reads as the airline charging the user for its problem.
  • Filters built around real questions. Earliest arrival is the goal for the family-event traveler. "Traveling with family" pre-checks group seat availability instead of letting it surprise the user three screens later.
  • Tap-target restated in the button. "Confirm 7:10 AM" tells you what you're confirming. Saves a back-tap on second-guessing.
  • [arrival time] is a bracketed placeholder. Real value is system-supplied at render. We don't invent a number.

Accessibility

Filter chips are real button elements with aria-pressed. Tabular numerals on flight times for readability. Each flight is an <article> with an accessible name.

Design notes

  • Coverage as a screen, not a footnote. The screen exists because hiding entitlements to reduce voucher cost is a trust-burning shortcut.
  • "Covered" / "Reimbursed" pills carry icon + label. Color is not the only signal. A color-blind user reads "Covered" the same way.
  • No invented voucher amount. "The amount is on your confirmation" is honest about where the value lives without inventing a number that policy hasn't blessed.
  • Visible opt-out. "Skip — I'm staying with family or friends" preserves user agency for the cohort that doesn't need a hotel.
  • Hotel is unnamed. Showing a specific hotel before policy says which one is a bait-and-switch waiting to happen.

Accessibility

Headings nested correctly inside the screen. Buttons name their action. Status pills do not rely on color alone (1.4.1, 1.4.11). Skip-button is keyboard reachable in the normal tab order — not hidden in a menu.

Screen 4

What happens next

The current confirmation says "Done." That's the worst possible word for the moment the user actually needs the most. This screen replaces it with a literal next-24-hours timeline, a save-offline primary, and a visible way to change plans again.

Design notes

  • A timeline, not "Done." "Done" is for the system. The user needs a chronological story.
  • Save offline is the primary. The user with one bar of signal at a gate needs the trip details in a place that doesn't require connectivity.
  • Share with someone is recovery for people traveling alone. The data is already in the app — the friction is only social.
  • "Change your booking again" is a visible link. Putting it in a footer or behind "Manage trip" is exactly the trick that makes people call instead.
  • Bracketed placeholders are deliberate. [gate], [new flight number], [time before flight] are system-supplied. We don't invent operational numbers — the design renders shaped slots so it's obvious where they go.

Accessibility

Ordered list semantics for the timeline (1.3.1). Bracketed placeholders are marked up so they read as "left bracket gate right bracket" to screen readers — they will be replaced before release. Save-offline action is keyboard reachable as the first action.

What this is. A four-screen mobile-first prototype of a canceled-flight recovery flow for Northstar Air (a fictional airline). It reframes the SMS, drops the misleading default, surfaces hotel and meal coverage, and replaces the dead-end confirmation with a save-to-offline timeline. The flow keeps within a 5-screen cap, doesn't invent airline policy or operational details, and supports light/dark modes plus reduced motion. WCAG-2.1-AA-relevant criteria are cited inline next to the decision they shaped: 1.3.1, 1.4.1, 1.4.3, 1.4.11, 2.1.1, 2.3.3, 2.4.7, 2.5.5, 4.1.3.
